## 問題一：美股池可用餘額對不上（NVDA 重複股息 + 遺漏利息）

使用者回報美股池「可用餘額」與證券戶實際美金現金對不上（App 顯示 $349.60，證券戶應為 $349.48）。

### 排查過程

1. **驗證公式本身**：手動用當日 CSV 匯出資料重算 `可用餘額 = 投入本金 + 已實現損益 + 已收股息 - 已投入成本`，逐日回溯多份歷史備份（06-26 → 07-01），確認公式邏輯自洽、無誤。
2. **抓出真正落差**：比對同一天前後兩次匯出的 CSV，發現 `[DIVIDENDS]` 多了一筆與既有記錄**完全相同**的 NVDA 除息（僅 `id` 與備注不同）：
   ```
   0f05e580-...,輝達,NVDA,USD,0.25,2,0.50,0.35,2026-06-29,NVDA除息   ← 原始
   be2e8db9-...,輝達,NVDA,USD,0.25,2,0.50,0.35,2026-06-29,          ← 重複
   ```
3. **使用者對照證券戶歷史交易**，找到一筆 6/19 美金帳戶利息 $0.23 未被記錄到工具中。
4. 驗算：`$349.25`（移除重複股息後）`+ $0.23`（補記利息）`= $349.48`，與證券戶完全吻合。

## 問題二：台股池可用餘額對不上（台達電賣出手續費分攤錯誤）

美股修正後，使用者接著回報台股池「可用餘額」也對不上：證券戶（扣除尚未交割的聯發科買入成本 $4,251 後）應為 $24,256，App 顯示 $24,251，差 $5。

### 排查過程

1. 精算 TWD 池每個組成（投入本金、已投入成本、已實現損益、股息），確認公式運算與 UI 顯示一致（$24,250.51 四捨五入 = $24,251），代表不是公式錯，是資料裡有問題。
2. 逐筆檢查 TWD 交易歷史，發現 06/29 台達電兩筆賣出（各賣 5 股，合計賣完原本 10 股、手續費 $5.50 的整批）的「買入手續費（攤分）」不一致：
   ```
   b280c356　5股　賣出價1930　買入手續費攤分 2.75  ← 正確
   646e1773　5股　賣出價1935　買入手續費攤分 8.25  ← 異常
   ```
3. 用程式驗證：10 股、手續費 $5.50 的批次依序切成兩筆 5 股賣出，正確攤分應該是 **2.75 + 2.75 = 5.50**（與原始手續費相等），而非 2.75 + 8.25 = 11.00。多算的 $5.50 手續費使該筆已實現損益被低估，進而拉低可用餘額。
4. 驗算：把已實現損益校正 +$5.50 後，`168,039 + 47,068.51 + 0 - 190,851.5 = $24,256.01`，與證券戶吻合。
5. 使用者確認「賣出手續費」「交易稅」欄位本身沒錯，且新增賣出的畫面**沒有**「買入手續費（攤分）」欄位可手動輸入——這個值是 `confirmSell()` 依 `buyFeeAlloc = 原手續費 × 賣出股數/當時剩餘股數` 自動算出的，不是誤打的。

### 根因（推測，無法完全還原歷史）
使用者稍早曾手動刪除過台達電的一筆重複賣出記錄（見下方問題一相關的雙擊送出成因）。`confirmSell()` 中 `buys.find(x=>x.id===sellTargetId)` 讀的是記憶體中的 `buys` 陣列快照；若當時因雙擊/連續操作導致多次呼叫，其中一次讀到的批次剩餘股數／手續費是還沒被前一次更新覆蓋的中間狀態，之後手動刪除「重複記錄」時，留下來的這筆分攤手續費就繼承了不正確的中間值（8.25），而不是乾淨路徑算出的 2.75。

## 根因（美股股息重複，適用於本文件問題一）

### A. 重複股息記錄
`confirmDividend()` 雖有「送出時禁用按鈕」防護（T9 已修過的雙擊防護），但這只能擋**同一次點擊被觸發兩次**，擋不住使用者**分兩次獨立送出**同一筆資料（例如網路不穩看不出是否存成功、又重新輸入一次）。兩次送出各自產生獨立的 `crypto.randomUUID()`，`upsert onConflict:'id'` 也無法偵測出這是重複。

### B. 重複資料難以被發現
`交易歷史`（sells）與`股息歷史`（dividends）表格原本用 `[...arr].reverse()` 排序，只是把陣列**插入順序**反轉，不是真的按 `sellDate`/`exDate` 排序。當補記舊資料、編輯、或如上述分兩次輸入時，同日期的記錄不會相鄰顯示，肉眼很難發現重複。

## 修正

1. **資料修正**（使用者手動執行）：
   - 刪除重複的 NVDA 股息記錄
   - 補記 6/19 美金利息 $0.23（資金明細 → 入金）
   - 編輯台達電 646e1773 賣出記錄，「買入手續費（攤分）」由 8.25 改為 2.75

2. **排序邏輯修正**（`index.html`）：
   - `交易歷史`：`[...sells].reverse()` → `[...sells].sort((a,b)=>b.sellDate.localeCompare(a.sellDate))`
   - `股息歷史`：`[...dividends].reverse()` → `[...dividends].sort((a,b)=>b.exDate.localeCompare(a.exDate))`
   - 與資金明細（T10 已正確實作 `sort by date desc`）邏輯一致，同日期記錄現在會相鄰顯示

3. **新增送出前重複檢查**（`index.html`）：
   - `confirmDividend()`：新增（非編輯）前，檢查 `dividends` 中是否已有**同標的 + 同除息日 + 金額相近**（誤差 < 0.005）的記錄，命中則跳確認對話框由使用者判斷
   - `confirmSell()`：新增（非編輯）前，檢查 `sells` 中是否已有**同標的 + 同賣出日 + 股數與價格相近**的記錄，命中則跳確認對話框
   - 不強制阻擋（同日多筆股息/分批賣出仍是合法情境），只多一層提醒

## 踩坑記錄

- **按鈕 disable 防護的邊界**：只能防「同一按鈕被觸發兩次」，防不了「使用者自己分兩次獨立操作」。這兩類是不同層級的問題，需要不同的防護手段（前者是 UI 層互斥鎖，後者是資料層重複比對）。
- **`.reverse()` ≠ 排序**：陣列反轉只在「插入順序恰好等於時間順序」時等價於時間排序，一旦有編輯、補登、離線同步等情境，這個假設就會被打破。新表格一律應該真正 `sort by date`，不要偷懶用 `reverse()`。
- **對帳排錯法**：先驗證公式本身（用歷史 CSV 逐日回溯），確認邏輯自洽後，才去找「資料」問題；避免把資料問題誤判成公式 bug。
- **分批賣出的手續費攤分是隱性狀態**：`buyFeeAlloc` 由程式自動算出、UI 上新增賣出時看不到這個欄位，只有事後「編輯」才會顯示。一旦上游批次的股數/手續費快照在雙擊或連續操作下出現中間不一致狀態，這個自動算出的值就會悄悄錯誤，且不會有任何錯誤提示——只能靠對帳才會發現。
- **多筆同批次分批賣出，加總應等於原始值**：驗證分批賣出資料是否正確的一個簡單方法，是把同一批次所有賣出的「買入手續費攤分」加總，應該精確等於該批次原始的 `buyFee`；股數同理。
